## 示例一：从排错对话中提炼知识

### 输入（用户粘贴的 Agent 对话节选）

```
用户：我的 Docker 容器启动后立刻退出，日志里只看到 "exec: \"python\": executable file not found in $PATH"

Agent：这个错误说明容器内找不到 Python 解释器。让我检查你的 Dockerfile...
看到了，你用的是 `FROM alpine:3.18` 作为基础镜像，但没有安装 Python。
Alpine 是最小化镜像，默认不带任何运行时。

我来修复：
1. 将基础镜像改为 `FROM python:3.11-slim`
2. 或者在 Alpine 基础上添加 `RUN apk add --no-cache python3`

推荐方案 1，因为 python:slim 已经配置好了 PATH 和入口点。

用户：换了之后报新错：ModuleNotFoundError: No module named 'flask'

Agent：依赖没有安装进镜像。你的 Dockerfile 里 COPY 了代码但没有 RUN pip install。
需要在 COPY 之后加上：
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

另外建议用多阶段构建减小镜像体积...
```

### 模式一提取结果

```
【上下文】Docker 化一个 Python Flask 应用
【问题/决策】容器启动失败，找不到 Python 解释器
【Agent 的做法】
  1. 识别错误信息 → 容器内无 Python
  2. 检查 Dockerfile → 发现用了 alpine 基础镜像
  3. 给出两个方案，推荐 python:slim
  4. 修复后出现新错误 → 依赖未安装
  5. 添加 pip install 步骤，建议多阶段构建
【底层原理】
  - Alpine 是最小化 Linux 发行版，默认不含任何运行时
  - Docker 镜像分层构建，每一层需要显式安装所需软件
  - 多阶段构建可以分离编译环境和运行环境，减小最终镜像体积
【替代方案】
  - 方案 A：用 python:3.11-slim（推荐，开箱即用）
  - 方案 B：在 alpine 上 apk add python3（更小但需手动配置）
  - 方案 C：用 python:3.11 完整版（最大但兼容性最好）
【可迁移教训】
  - 选择基础镜像时，"最小"不等于"最好"，要权衡便利性和体积
  - 容器报错第一时间看日志，日志是排查的第一手信息
  - Docker 构建是声明式的，每个依赖都要显式声明
```

### 模式二生成的面试题

```
### [Bug 排查题] Docker 容器启动后立即退出

**难度**：基础
**标签**：#Docker #容器化 #排错
**来源**：基于 Flask 应用容器化的部署阶段提取

#### 场景描述
你刚把一个 Python Web 应用容器化，执行 docker run 后容器立刻退出了。
docker logs 显示 "executable file not found in $PATH"。

#### 问题
你的排查思路是什么？如何修复？

#### 参考思路
1. 首先看 docker logs 的具体错误信息
2. 分析 "executable not found" → 运行时环境缺失
3. 检查 Dockerfile 的 FROM 指令 → 基础镜像是否包含所需运行时
4. 修复后如果有新错误，继续迭代排查（依赖安装、端口映射等）

#### 反事实推理
- 如果这个应用是 Node.js 的，排查思路有什么不同？
- 如果你必须用 Alpine 镜像（公司安全策略），怎么处理？
- 如果容器不是立刻退出，而是运行 5 分钟后退出，排查方向怎么变？

#### 深挖追问
- Docker 镜像的分层机制是什么？为什么层数越少越好？
- 多阶段构建的原理是什么？它如何减小最终镜像体积？
- slim 镜像和 alpine 镜像的本质区别是什么？
```

### 模式三沉淀的避坑条目

```
### [坑-001] Alpine 基础镜像缺少运行时环境

**领域**：部署
**严重程度**：中
**来源**：Flask 应用容器化 - 2026-06-24

#### 现象
Docker 容器启动后立即退出，日志报 "executable file not found in $PATH"

#### 根本原因
Alpine 是最小化 Linux 发行版，默认不包含 Python/Node 等运行时。
开发者常因"镜像小"选择 Alpine，但忽略了它需要手动安装所有依赖。

#### 解决过程
1. 看日志 → 发现找不到 Python
2. 检查 Dockerfile → 发现 FROM alpine
3. 更换基础镜像为 python:slim → 解决
4. 出现新错误 → 依赖未安装 → 添加 pip install

#### 解决方法
将基础镜像从 alpine 换为对应运行时的 slim 版本（如 python:3.11-slim）

#### 预防措施
- 选择基础镜像时，先确认包含所需运行时
- 本地先 docker build && docker run 验证，不要直接推到 CI
- 在 Dockerfile 注释中标注选择该基础镜像的理由
```

## 示例二：从架构设计对话中提炼知识

### 模式二生成的面试题

```
### [方案对比题] 为什么用 PostgreSQL + Neo4j 双数据库架构？

**难度**：进阶
**标签**：#架构 #数据库 #知识图谱
**来源**：基于 MAD-EduKG 项目的架构设计阶段提取

#### 场景描述
你正在构建一个教育知识图谱系统，需要存储实体、关系，也需要做图遍历查询。
团队有人提议"全部用 Neo4j，一个数据库搞定"。

#### 问题
你会怎么分析这个提议？最终如何选择？

#### 参考思路
1. 分析数据访问模式：哪些是 CRUD 操作？哪些是图遍历？
2. 评估技术成熟度：团队对两种数据库的熟悉程度
3. 考虑运维成本：维护两个数据库 vs 一个数据库的复杂度
4. 设计数据同步策略：如果选双库，如何保证一致性？

#### 反事实推理
- 如果团队所有人都熟悉 Neo4j，还会选双库方案吗？
- 如果图查询只占 5% 的请求，双库的运维成本还值得吗？
- 如果用 PostgreSQL 的扩展（如 Apache AGE）做图查询，能否替代 Neo4j？

#### 深挖追问
- Neo4j 的 Cypher 查询语言和 SQL 的本质区别是什么？
- 两个数据库之间的数据同步，你用的什么策略？如何保证一致性？
- 如果同步延迟了，用户体验怎么保证？
```

## 示例四：安装与跨平台使用

### 最简安装（一句话）

直接把这句话发给你的 AI Agent：

```
帮我安装这个 Skill：https://github.com/bb-cccc/vibe-upskill
```

Agent 会自动完成克隆、放置、验证全流程。适用于 Claude Code、Cursor、Copilot 等任何支持 Skill/插件机制的 Agent。

### 手动安装

本 Skill 设计为通用格式，可在多种 AI Agent 中使用：

**Claude Code**：将 `vibe-upskill/` 目录放在项目的 `.claude/skills/` 下，Claude Code 会自动识别 SKILL.md 并在合适时机调用。

**Codex / 其他 Agent**：将 SKILL.md 放在项目根目录或 `.agent/skills/` 下，在对话中提及"复盘"、"面试题生成"等触发词即可激活。

**通用用法**：在任何 AI Agent 对话中，直接粘贴 SKILL.md 的内容作为系统指令，然后提供你的开发对话记录即可开始复盘。
